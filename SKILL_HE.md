# מתעד קוד (Code Annotator)

מוסיף הערות מסבירות לקוד בלי לשנות אף שורת לוגיקה. המטרה: הערות שמפתח חדש יכול לקרוא ולהבין **מה** הקוד עושה, **למה** הוא עושה זאת כך, ו**איך** הוא מתחבר לשאר ה-codebase — לא הערות שסתם חוזרות על התחביר.

## כללי הזהב (לא לשבור לעולם)

1. **לעולם לא לשנות את התנהגות הקוד.** רק להוסיף או לשפר הערות ורווחים סביבן. לא לשנות שמות, לא לסדר מחדש, לא "לתקן". אם מזהים באג אמיתי — לציין אותו למשתמש בנפרד, לא לשנות אותו בשקט.
2. **לעולם לא למחוק קוד עובד.** הערות הן תוספת בלבד.
3. **לא להעיר על המובן מאליו.** `i++ // מגדיל את i` זה רעש. מעירים על כוונה, חוזים, והחלטות לא טריוויאליות — לא על תחביר מובן מאליו.
4. **לשמר ולשדרג הערות קיימות.** אם כבר יש הערה, שומרים על משמעותה; משפרים בהירות רק אם היא שגויה או לא ברורה. לא לשכפל.
5. **להתאים לתחביר ההערות של הקובץ.** `//` ל-Dart/TS/JS/PHP/C++, `#` ל-Python/Ruby/shell, `<!-- -->` ל-HTML, `///` ל-doc comments ב-Dart במקומות המתאימים.

## בחירת שפת ההערות (עברית מול אנגלית)

מחליטים לכל קובץ, בסדר הזה:
1. אם בקובץ כבר יש הערות — **מתאימים לשפה שלהן**.
2. אם אין הערות — משתמשים בשפה שהמשתמש ביקש בבקשה.
3. אם עדיין לא ברור — שואלים את המשתמש פעם אחת: "עברית או אנגלית להערות?" ומיישמים את התשובה על כל המשימה.

שומרים מזהים בקוד, שמות API ומחרוזות שגיאה בדיוק כמו שהם — רק הטקסט הפרוזאי של ההערות מתורגם.

## תהליך העבודה

### שלב 1 — להבין לפני שכותבים
לפני שמעירים משהו, בונים מפה מנטלית:
- קוראים את הקובץ/ים במלואם.
- עוקבים אחרי החיבורים: מה הקובץ מייבא/דורש? לאילו פונקציות הוא קורא במקומות אחרים? מי קורא **אליו**? (grep לשמות הפונקציות/מחלקות בכל ה-codebase.)
- מזהים את החוזים: סדר פרמטרים שמשנה, מבני return שקוד אחר מסתמך עליהם, תופעות לוואי, סדר קריאה נדרש, הנחות auth/אבטחה, state גלובלי שנוגעים בו.

רק אחרי שמבינים איך הקובץ משתלב בשלם, מתחילים לכתוב הערות. זה מה שמבדיל את הסקיל הזה מתיאור שורה-אחר-שורה.

### שלב 2 — כותרת קובץ
בראש כל קובץ, מוסיפים בלוק כותרת:

```
// ─── <שם הקובץ> — <שורת מטרה אחת בשפה הנבחרת> ──────────────────────
// <1–3 שורות: על מה הקובץ אחראי>
// <חיבורים: מי קורא לו / למי הוא קורא — למשל "נקרא על ידי updateTicketStatus() ב-admin_dashboard.js">
```

משתמשים בקווי box-drawing (`─`, U+2500) כדי לרפד את המחיצה לרוחב אחיד (~78 תווים). שומרים על מראה מסודר ומיושר.

### שלב 3 — מחיצות סקופ / סקציה
מפצלים את הקובץ לסקופים לוגיים ומסמנים כל אחד עם מחיצה בעלת שם:

```
// ─── <שם הסקופ> ───────────────────────────────────────────────────────────
// <למה הסקופ הזה קיים / מה הוא מקבץ>
```

דוגמאות לסקופים: "בדיקת method", "איסוף פרמטרים", "עדכון בDB", "Helpers", "הגדרת State", "Render". מקבצים לפי כוונה, לא לפי מכניקה.

### שלב 4 — הערות לכל פונקציה
מעל כל פונקציה/מתודה, מסבירים:
- **מה** היא עושה (שורה אחת, ברמת הכוונה).
- **למה / מתי** משתמשים בה, אם לא מובן מאליו.
- **הערות חוזה** שהקורא חייב לדעת: מלכודות בסדר הפרמטרים, תנאים מוקדמים נדרשים, מה היא מחזירה, תופעות לוואי, מה קורה בשגיאה.
- **הצלבות**: מי קורא לזה, ועל מה זה תלוי. למשל `// נקרא על ידי TaskRepository.submit() — מצפה ל-userId כבר מאומת`.

### שלב 5 — הערות חוזה ומלכודות בתוך הקוד
בתוך פונקציות, מוסיפים הערות קצרות רק במקום שבו החלטה אינה מובנת מאליה או נושאת משקל. סדר עדיפויות:
- נימוק אבטחה ("Prepared Statement מונע SQL Injection — הערכים מועברים בנפרד מהשאילתה").
- למה שגיאה נחשפת / לא נחשפת לגולש.
- סדר פרמטרים שחייב להתאים למשהו אחר ("סדר הפרמטרים תואם את הסדר של ? בשאילתה").
- הנחות לגבי state, auth, או סדר קריאה שאם מופרות — שוברות דברים.

### שלב 6 — מעבר ביקורת
אחרי ההערות, קוראים את הקובץ מחדש כאילו אתם רואים אותו בפעם הראשונה. מוחקים כל הערה שרק חוזרת על הקוד. מוודאים: האם נגעתם בלוגיקה כלשהי? אם כן — מחזירים לאחור.

## דוגמת סגנון (פלט מטרה)

זו רמת האיכות שאליה שואפים. הדבר המרכזי לשים לב אליו: ההערות הופכות את **האינטגרציה בין הקבצים למפורשת**. הכותרת של כל קובץ מציינת עם מי הוא מדבר, והערות החוזה נוקבות בשם המדויק של הפונקציה או המחלקה בצד השני — כך שהקורא אף פעם לא צריך לנחש איך החלקים מתחברים. הדוגמה למטה היא שלושה קבצים של עגלת קניות קטנה: פונקציית עזר משותפת, הקוד שמשתמש בה, וגיליון הסגנון שחולק חוזה של שם-מחלקה עם אותו קוד.

**קובץ 1 — `format-price.js` (פונקציית עזר משותפת שקבצים אחרים תלויים בה):**

```js
// ─── format-price.js — עזר לפורמט מטבע ──────────────────────────────────────
// פונקציה טהורה, ללא תופעות לוואי. הופכת סכום שלם ביחידות משנה
// (סנט / אגורות) למחרוזת מחיר מקומית.
// נמצא בשימוש: cart.js → renderTotal(). אם משנים כאן את מבנה ההחזרה,
// צריך לעדכן גם את ה-assertion ב-cart.test.js.

export function formatPrice(minorUnits, currency = 'USD') {
  // עובדים ביחידות משנה (מספרים שלמים) כדי להימנע משגיאות עיגול של נקודה צפה.
  // הקוראים חייבים להעביר סנטים שלמים — העברת 9.99 במקום 999 שגויה פי 100.
  const major = minorUnits / 100;
  return new Intl.NumberFormat('en-US', { style: 'currency', currency }).format(major);
}
```

**קובץ 2 — `cart.js` (צורך את העזר שלמעלה, וחולק חוזה עם ה-CSS):**

```js
// ─── cart.js — רינדור עגלת קניות ────────────────────────────────────────────
// הבעלים של ה-DOM של העגלה: קורא state, מרנדר שורות, צובע את הסכום.
// תלוי ב: format-price.js (formatPrice).
// חולק חוזה עם cart.css: מחליף את המחלקה .cart--empty — שם המחלקה חייב
// להישאר זהה בשני הקבצים, אחרת עיצוב מצב-העגלה-הריקה נשבר.

import { formatPrice } from './format-price.js';

// ─── סכום ────────────────────────────────────────────────────────────────────
function renderTotal(items) {
  // הסכום נשאר בסנטים — formatPrice מצפה ליחידות משנה, ולכן לא מחלקים כאן.
  const sum = items.reduce((acc, item) => acc + item.priceCents * item.qty, 0);

  // formatPrice() נמצא ב-format-price.js — מעבירים סנטים גולמיים, לא דולרים.
  totalEl.textContent = formatPrice(sum);

  // .cart--empty נצרך על ידי cart.css כדי להסתיר את כפתור ה-checkout.
  // משנים את השם כאן — חייבים לשנות גם שם, אחרת שבירה שקטה.
  cartEl.classList.toggle('cart--empty', items.length === 0);
}
```

**קובץ 3 — `cart.css` (הערות על קובץ שאינו קוד: הסקיל מבין את כל הפרויקט, לא רק לוגיקה):**

```css
/* ─── cart.css — סגנונות עגלת הקניות ─────────────────────────────────────────
   מעצב את ה-DOM שנבנה על ידי cart.js. שמות המחלקות כאן הם חוזה עם ה-JS:
   cart.js מחליף את .cart--empty בזמן ריצה, ולכן שינוי שמו כאן שובר בשקט
   את התנהגות מצב-הריק. שומרים את שני הקבצים מסונכרנים. */

/* ─── מצב ריק ─────────────────────────────────────────────────────────────── */
/* מופעל כש-cart.js מגדיר .cart--empty (items.length === 0).
   מסתיר את כפתור ה-checkout כדי שלא ניתן יהיה לשלוח עגלה ריקה. */
.cart--empty .checkout-btn {
  display: none;
}
```

## הערות לפי שפה

- **Dart / Flutter**: להשתמש ב-`///` doc comments מעל מחלקות, widgets ו-providers ציבוריים כדי שיופיעו ב-tooltips של ה-IDE; `//` למחיצות סקופ פנימיות והערות inline. ל-providers של Riverpod — לציין מה ה-provider חושף ומי צופה בו.
- **TypeScript / JS (למשל Supabase Edge Functions)**: לציין את מבנה ה-request/response, הנחות auth (מי חייב להיות מאומת), וטיפול ב-CORS. להפנות ל-helpers משותפים בשמם.
- **Python**: הערות `#` לסקופים; לשקול docstring קצר `"""..."""` לכותרת הקובץ במקום בלוק `#`.

## היקף ריצה

- אם המשתמש מצביע על **קובץ אחד**, מעירים עליו במלואו.
- אם המשתמש אומר "כל הפרויקט", עובדים קובץ-קובץ. מאשרים את הסדר או מתחילים מהקבצים המרכזיים ביותר (אלה שהכי הרבה תלויים בהם). מסכמים התקדמות כדי שהמשתמש יוכל לעצור בכל רגע.
- תמיד מראים למשתמש את הקובץ/ים המוערים ונותנים לו לאשר את הסגנון לפני יישום המוני על קבצים רבים.
